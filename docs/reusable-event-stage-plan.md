# 可复用现场大屏方案

## 判断

当前页面的问题不是功能缺少一点，而是「控制台」和「现场大屏」混在了一起。

控制台应该服务操作员：选人、抽题、开 PPT、看同步状态。

现场大屏应该服务观众和主持人：收拢注意力、建立开场氛围、解释当前环节、放大关键瞬间、在信息之间留出节奏。

所以后续不要把投屏页当作控制台镜像，而要把它做成一个可被控制台驱动的舞台状态机。

## 目标

以后复用到其他活动时，代码不变，只替换一份飞书多维表格或更新 Vercel 的飞书表格配置。

活动方只需要维护内容：

- 活动标题、副标题、城市、日期、开场文案
- 报名者信息
- 选手/观众身份
- 三个关键词
- Demo 或作品描述
- 题库和 PPT 文件名
- 可选的流程段落

页面负责：

- 自动读取飞书
- 清洗字段
- 生成控制台
- 驱动投屏状态
- 保存抽题记录
- 输出可复盘日志

## 信息架构

建议把飞书 Base 做成一个可复制模板，至少三张表。

### 1. 活动配置

表名：`活动配置`

字段建议：

- `key`
- `value`
- `note`

常用 key：

- `event_title`
- `event_subtitle`
- `city`
- `date`
- `venue`
- `opening_line`
- `standby_hint`
- `closing_line`
- `brand_name`
- `theme`
- `mascot`

这样杭州活动可以是「兜来乱讲 PPT」，以后换成别的活动只改这张表。

### 2. 报名者

表名：`报名者` 或直接使用问卷结果表。

标准语义字段：

- `nickname`
- `presence`
- `role`
- `keywords`
- `demo_description`
- `demo_url`
- `is_speaker`
- `display_status`
- `checkin_status`
- `display_order`

页面不应该强依赖中文题目完全一致，而是用字段 ID 和别名表映射到这些标准语义字段。

### 3. 题库

表名：`题库`

字段建议：

- `topic_id`
- `topic_title`
- `topic_summary`
- `ppt_no`
- `ppt_file`
- `enabled`
- `weight`

如果某个活动不需要抽题，这张表可以为空；大屏进入纯展示模式。

### 4. 流程，可选

表名：`流程`

字段建议：

- `segment_id`
- `mode`
- `title`
- `subtitle`
- `duration_seconds`
- `visible`
- `sort`

这个表用于更复杂的活动，比如开场、嘉宾介绍、中场、Demo、结束照。

## 页面状态机

现场大屏至少要有这些状态。

### Standby

开场待机。用于观众入场、聊天、等待主持人开始。

显示：

- 活动标题
- 活动副标题
- 报名人数、选手人数、线上/线下人数
- 关键词墙
- 吉祥物或主视觉

不显示具体某个人，避免一开屏就显得混乱。

### Focus

开场聚焦。主持人准备说话时使用。

显示：

- 大标题
- 一句开场文案
- 可选 10 秒倒计时

作用是把现场注意力从闲聊拉回大屏。

### Lobby

候场。可以轮播报名者、关键词、今晚流程。

显示：

- 当前环节
- 下一位或候选队列
- 关键词云

### Caller

叫号。主持人点到某人时使用。

显示：

- 当前报名者昵称
- 身份：选手/观众
- 参与方式：线上/线下
- 三个关键词
- Demo 描述摘要

### Draw

抽题揭晓。不能只是静态变字，要有一个明显的 reveal 瞬间。

显示：

- 正在抽题
- 锁定题目
- PPT 编号
- 题目摘要

### Talk

展示中。真正进入选手讲述时，大屏应降噪。

显示：

- 选手名
- 题目名
- 计时器
- PPT 编号

不要继续塞报名表细节。

### Intermission

中场。用于等待下一位、休息、同步数据。

显示：

- 已完成人数
- 未完成队列
- 关键词墙
- 现场提示

### Closing

结束。用于收尾和拍照。

显示：

- 今晚参与者墙
- 已完成选手
- 活动 closing line
- 可选二维码或后续链接

## 控制台逻辑

控制台不应该只靠选人驱动投屏，还应该有一个「投屏模式」控制条。

建议按钮：

- `开场`
- `聚焦`
- `候场`
- `叫号`
- `抽题`
- `展示`
- `中场`
- `结束`

控制台状态建议保存为：

```json
{
  "stageMode": "standby",
  "currentParticipantId": "rec...",
  "pendingDrawParticipantId": null,
  "draws": {},
  "timer": {
    "status": "idle",
    "startedAt": null,
    "durationSeconds": 300
  },
  "updatedAt": "..."
}
```

投屏页只读这个状态并渲染，不直接承担复杂操作。

## 复用策略

短期最稳的复用方式：

1. 复制飞书 Base 模板
2. 替换报名表或结果表
3. 在 Vercel 更新这几个环境变量：
   - `FEISHU_BITABLE_APP_TOKEN`
   - `FEISHU_BITABLE_TABLE_ID`
   - `FEISHU_BITABLE_VIEW_ID`
   - `FEISHU_TOPIC_TABLE_ID`
   - `FEISHU_CONFIG_TABLE_ID`
4. 页面自动同步，无需改代码

中期更顺手的方式：

- 做一个 `events` 配置表
- 每个活动一个 `slug`
- URL 使用 `?event=hangzhou-vibecoding`
- 后端按 slug 找对应 Base 和 table

这样以后甚至不用改 Vercel 环境变量，只需要在配置表新增一行活动。

## 实施顺序

第一阶段：把当前杭州页补成可用现场大屏。

- 删除无意义装饰
- 增加投屏模式状态
- 增加开场待机页
- 增加关键词墙
- 增加控制台模式切换

第二阶段：把活动内容从代码里拿出去。

- 增加 `eventConfig`
- 飞书读取活动配置表
- 页面标题、文案、统计、closing 都从配置读取

第三阶段：做成可复用模板。

- 固定飞书模板字段
- 保留字段别名兼容
- 输出一份复制说明
- 增加缺字段提示

第四阶段：增强现场能力。

- 抽题 reveal 动画
- Talk 计时器
- Closing 参与者墙
- 抽题记录写回飞书

## 关键原则

大屏不是数据库视图。

大屏只在正确时刻显示正确的信息。开场时聚焦，叫号时放大人，抽题时放大结果，展示时降噪，结束时收束。

只要这个原则成立，换活动就只是换数据，而不是重新设计一套页面。
